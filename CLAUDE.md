## Project Overview

Hiddify is a multi-platform proxy client built with Flutter, based on the [Sing-box](https://github.com/SagerNet/sing-box) universal proxy tool-chain. It supports Android, iOS, Windows, macOS, and Linux.

## Build & Development Commands

### Initial Setup (per platform)
```bash
make windows-prepare     # Windows: pub get + codegen + translations + core libs
make linux-amd64-prepare # Linux AMD64
make android-prepare     # Android
make ios-prepare         # iOS (includes pod install)
make macos-prepare       # macOS
```

### Common Development Tasks
```bash
make get                 # flutter pub get
make gen                 # dart run build_runner build --delete-conflicting-outputs
make translate           # dart run slang (regenerate translations)
make common-prepare      # get + gen + translate (no native libs)
flutter run --device-id=<id>  # Run locally on a target device
```

### Running Tests
```bash
flutter test             # Run all tests (CI baseline: make linux-amd64-prepare && flutter test)
flutter test test/path/to/test_file.dart  # Run a single test
```

Test files mirror source paths: `lib/features/profile/...` → `test/features/profile/...`. Existing tests cover DB migrations (`test/drift/migration_test.dart`), profile parsing (`test/features/profile/`), and core utilities (`test/core/`). Add DB migration tests when changing the schema.

### Building Releases
```bash
make android-apk-release   # Build Android APK
make windows-release       # Build Windows (zip + exe + msix)
make linux-release         # Build Linux (deb + appimage)
make macos-release         # Build macOS (dmg + pkg)
```

### Entry Points
- `lib/main.dart` — Development entry point
- `lib/main_prod.dart` — Production entry point (used when `CHANNEL=prod`)

## Architecture

### State Management: Riverpod
- Uses `hooks_riverpod` with `@Riverpod` annotation-based code generation
- Providers are generated into `.g.dart` files via `riverpod_generator`
- `flutter_hooks` for composable reactive widgets
- `rxdart` for complex async stream operations

### Feature-First Structure (`lib/features/`)
Each feature module follows a layered pattern:
- `data/` — Repositories and data providers
- `notifier/` — Riverpod state notifiers
- `model/` — Domain entities (often using `freezed`)
- `widget/` — UI components

Key features: `connection/` (proxy engine control), `profile/` (VPN profile management), `settings/`, `home/`, `deep_link/`, `system_tray/` (desktop), `per_app_proxy/` (Android), `window/` (desktop multi-window), `route_rules/` (routing rules), `stats/` (connection statistics).

### Core Layer (`lib/core/`)
Shared infrastructure: `db/` (Drift/SQLite), `router/` (GoRouter with deep linking), `theme/` (Material 3), `http_client/` (Dio), `localization/`, `analytics/` (Sentry), `preferences/` (SharedPreferences), `logger/` (Loggy), `notification/`.

### Native Core Integration (`lib/hiddifycore/`)
- Go-based proxy engine (hiddify-core) integrated via FFI on desktop and gRPC/protobuf on mobile
- `core_interface/` — Platform-split interfaces: `core_interface_desktop.dart`, `core_interface_mobile.dart`, with `core_interface_wrapper.dart` and a dev stub
- `generated/` — Protobuf definitions (extension service, common v2 types)
- FFI bindings generated by `ffigen` from `hiddify-core/bin/desktop.h` → `lib/gen/hiddify_core_generated_bindings.dart`
- Prebuilt native libraries downloaded via Makefile targets (`*-libs`)

### Platform-Native Code
- **Android** (`android/.../kotlin/`): `bg/BoxService.kt` (background proxy service), `BootReceiver.kt`, `AppChangeReceiver.kt`, `DefaultNetworkMonitor.kt`, `ActiveGroupsChannel.kt` (Flutter platform channel), `PlatformInterfaceWrapper.kt`
- **iOS** (`ios/Runner/`): `AppDelegate.swift`, event handlers for active groups/alerts/logs/methods, VPN extension in `Runner/VPN/`, Network Extension in `HiddifyPacketTunnel/`
- **Windows** (`windows/runner/`): Win32 window management (`win32_window.cpp`, `flutter_window.cpp`), `main.cpp`

### Generated Code (`lib/gen/`)
Do not edit files in this directory. They are generated by:
- `build_runner` — Freezed models, JSON serialization, Drift schemas, Riverpod providers (`.g.dart`, `.freezed.dart`)
- `slang` — `translations.g.dart` from `assets/translations/*.i18n.json`
- `ffigen` — `hiddify_core_generated_bindings.dart`
- `flutter_gen_runner` — `assets.gen.dart`

### Database
- ORM: Drift (type-safe SQLite)
- Config in `build.yaml`: schema dir is `lib/core/db/schemas/`, DateTimes stored as text
- Main DB class: `lib/core/db/db.dart`
- Schema versions 1–5 exist in `test/drift/db/generated/`; always add a migration test when bumping the schema version

### Error Handling
Mixin-based hierarchy: `Failure` → `UnexpectedFailure` / `ExpectedFailure` / `ExpectedMeasuredFailure`. Errors are localized via `ErrorPresenter` extension and tracked with Sentry.

## Code Style & Linting

- Strict linting via `package:lint/strict.yaml`
- Formatter page width: **120 characters**
- Generated files excluded from analysis: `**.g.dart`, `lib/gen/**`, `hiddify-core/**`
- Custom lint rule: `provider_parameters`
- Naming: `snake_case.dart` files, `PascalCase` types/enums, `camelCase` methods/variables

## Localization

- Framework: `slang` + `slang_flutter`
- Base locale: `en`
- Translation files: `assets/translations/*.i18n.json`
- Output: `lib/gen/translations.g.dart`
- Run `make translate` or `dart run slang` after modifying translation files

## Commit Conventions

Recent history uses concise prefixes: `fix:`, `new:`, `release:`, `chore:`. Keep commits focused (one logical change per commit). PRs should include change summary, motivation, linked issue, platforms tested, and screenshots/video for UI-visible changes.
